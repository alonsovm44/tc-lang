# Tig 1.2.3 Full Documentation

## Overview

Tig 1.2.0 introduces several major features that significantly enhance the language's capabilities for systems programming:

- **Varargs support** ‚Äî Functions can now declare and use variadic arguments with `...`
- **Expanded I/O stdlib** ‚Äî Added comprehensive file I/O functions
- **Else if statements** ‚Äî Added `else if` and `else` for chained conditionals
- **Enums** ‚Äî Simple enums for type-safe constants
- **Match statements** ‚Äî Pattern matching with wildcard support
- **Pointer field access** ‚Äî `.>` syntax for accessing fields through pointers
- **Pass-by-reference** ‚Äî `&` operator for passing pointers to functions
- **Methods in struns** ‚Äî Object-oriented methods with automatic `self` parameter
- **Inline C code** ‚Äî Direct C code injection using `"C"{ ... }` syntax

---

## Varargs Support

Functions can now declare variadic arguments using `...`:

```tc
extern "C" {
    i32 fn printf: ->i8 fmt, ... {}
}
```

### Usage

When calling varargs functions, you don't need to write `...` in the call - it's implicit:

```tc
use "stdlib/io.tc"

extern "C" {
    i32 fn printf: ->i8 fmt, ... {}
}

fn void main: {
    printf("Hello %s, number: %d\n", "world", 42)
}
```

### Implementation Notes

- `...` in function declarations is emitted as `...` in the generated C function signature
- `...` in function calls is not emitted - varargs are implicit in C function calls
- The compiler tracks varargs with the `EX_VARARGS` AST node

## Expanded I/O Stdlib

### Standard I/O Functions

| Function | Description |
|----------|-------------|
| `print(s)` | Print string + newline |
| `printn(s)` | Print string, no newline |
| `printi(n)` | Print i64 + newline |
| `printin(n)` | Print i64, no newline |
| `readi()` | Read i64 from stdin |
| `readc()` | Read single char from stdin |
| `unreadc(c, stream)` | Push char back to file stream |
| `write_file(s, stream)` | Write string to file |
| `eof(stream)` | Check if at end of file |

### File I/O (via extern "C")

| Function | Description |
|----------|-------------|
| `fopen(file, mode)` | Open file |
| `fclose(f)` | Close file |
| `fgetc(stream)` | Read char from file |
| `ungetc(c, stream)` | Push char back to file |
| `fputs(s, stream)` | Write string to file |
| `fprintf(stream, fmt, ...)` | Formatted print to file |
| `fscanf(stream, fmt, ...)` | Formatted read from file |
| `feof(stream)` | Check end of file |

### Example

```tc
use "stdlib/io.tc"

extern "C" {
    ->FILE fn fopen: ->i8 file, ->i8 mode {}
    i32 fn fclose: ->FILE stream {}
    i32 fn fputs: ->i8 s, ->FILE stream {}
}

fn void main: {
    ->FILE f = fopen("test.txt", "w")
    if (f) {
        fputs("Hello, file!\n", f)
        fclose(f)
    }
}
```

## Else If Statements

Tig 1.2.0 introduces `else if` for chained conditionals, with `else` as the final branch:

```tc
fn void check_grade: i32 score {
    if (score >= 90) {
        print("A")
    } else if (score >= 80) {
        print("B")
    } else if (score >= 70) {
        print("C")
    } else if (score >= 60) {
        print("D")
    } else {
        print("F")
    }
}
```

### Syntax Rules

- `else if (condition) { ... }` ‚Äî else-if branch
- `else { ... }` ‚Äî final else branch (must be last)
- The `_` wildcard is now reserved for match statements only

## Enums

Tig 1.2.0 introduces simple enums for type-safe constants:

```tc
enum Color {
    RED
    GREEN
    BLUE
    BLACK
    WHITE
}

fn void main: {
    Color c = RED
    if (c == RED) {
        print("Color is RED\n")
    }
}
```

### Syntax Rules

- `enum Name { MEMBER1, MEMBER2, ... }` ‚Äî define an enum
- Enum members are scoped to the enum type
- Types default to i32 (C doesn't support enum type specification like C++)
- Generated as `typedef enum { ... } Name;` in C

### Usage

Enums can be used as variable types and in comparisons:

```tc
enum Status {
    ACTIVE
    INACTIVE
    PENDING
}

fn void set_status: Status s {
    if (s == ACTIVE) {
        print("Status is active\n")
    }
}
```

## Match Statements

Pattern matching with wildcard support:

```tc
fn void describe: i32 n {
    match (n) {
        1 = {
            print("one")
        }
        2 = {
            print("two")
        }
        3 = {
            print("three")
        }
        _ = {
            print("other")
        }
    }
}
```

### Syntax

- `match (expr) { ... }` ‚Äî match expression against patterns
- `pattern = { ... }` ‚Äî pattern branch
- `_ = { ... }` ‚Äî wildcard (catch-all) branch
- Patterns are currently single values (literals or variables)

## Pointer Field Access (.>)

To avoid ambiguity with the `->` operator (used for pointer declaration and dereference), Tig uses `.>` for accessing fields through pointers:

```tc
strun Person {
    i8 age
    ->i8 name
}

void fn foo: ->Person p {
    p.>age = 23
    p.>name = "John"
}

void fn main: {
    Person p
    foo(&p)
    printi(p.age)
    print(p.name)
}
```

### Generated C

```c
void foo(struct Person *p) {
    p->age = 23;
    p->name = "John";
}

void main(void) {
    struct Person p = {0};
    foo(&p);
    printi(p.age);
    print(p.name);
}
```

### Alternative Syntax

You can also dereference explicitly:

```tc
void fn boo: ->Person p {
    (->p).age = 23
    (->p).name = "John"
}
```

## Pass-by-Reference (&)

Use `&` to pass pointers to functions:

```tc
void fn modify: ->i32 x {
    ->x = 100
}

fn void main: {
    i32 value = 10
    modify(&value)
    printi(value)  // Prints 100
}
```

### Generated C

```c
void modify(int32_t *x) {
    *x = 100;
}

void main(void) {
    int32_t value = 10;
    modify(&value);
    printi(value);
}
```

## Global Hot Reloading (v1.2.3)

Tig supports global, bulletproof hot reloading. This allows you to modify functions, structs (`strun` definitions), and enums, and recompile shared libraries *completely on-the-fly* without restarting your running application. No special `hot` keyword is needed anymore!

### Basic Usage

```tc
use "stdlib/io.tc"

extern "C" {
    i32 fn Sleep: u32 ms {}
}

fn i32 add: i32 x, i32 y {
    ret x + y
}

fn i32 main: {
    loop {
        i32 result = add(3, 4)
        printi(result)
        Sleep(2000)
    }
}
```

1. **Compile with hot reload enabled**:
```bash
tigc source.tc -H hotlib -c app
```

2. **Run the application**:
```bash
./app
```

3. **While running, modify the source code (e.g. change add to multiply) and rebuild**:
```bash
tigc source.tc -H hotlib --hot
```

The running application automatically detects the change, loads the new code, and continues running seamlessly!

### How It Works

Tig uses a robust Host/DLL splitting architecture to avoid Windows file locking issues and guarantee clean reloads during loops:

- **Host (The Driver)**: The `main` function is compiled directly into the host executable. This ensures the main driver/application loop resides safely outside the shared library, avoiding trapped call stacks. The host manages loading/unloading and resolves stubs.
- **Library (The Engine)**: All other functions, structs, and enums are compiled into the shared library (`hotlib_N.dll` on Windows / `hotlib_N.so` on Unix). Every function is exported automatically.
- **Dynamic Reloading**: When a function is called, a host stub checks the current library version, reloads if a new version is detected, and executes through function pointers.
- **Automatic Cleanup**: On a successful reload, the host automatically and cleanly deletes old version files to keep your workspace pristine.

### Additional Flags

| Flag | Description |
|------|-------------|
| `-H <libname>` | Enable hot reload mode and specify the shared library name |
| `--hot` | Rebuild only the hot library version for a running application |
| `-t, --temp` | Keep temporary `.c` files for debugging |

## Strunions (Layout Polymorphism)

Strunions combine structs and unions into a single construct:

```tc
// Regular struct
strun Point {
    i32 x
    i32 y
}

// Union
strun Data {
    &i32 data
    &str ip
}

// Hybrid
strun Hybrid {
    i32 x
    i32 y
    &i32 z
    &f32 w
}
```

- Regular fields -> struct behavior (separate memory)
- `&`-prefixed fields -> union behavior (shared memory)
- Anonymous padding for alignment control

## Methods in Struns

Struns can now contain methods, which are functions that operate on the struct instance. Methods provide object-oriented programming capabilities while maintaining the simplicity of C.

### Method Syntax

```tc
strun Person {
    i32 age
    i8 name
    
    // Method without parameters
    fn void sayHi: {
        print("Hi from Person!\n")
    }
    
    // Method with parameters
    fn void setAge: i32 newAge {
        self.>age = newAge
    }
    
    // Method with return value
    fn i32 getAge: {
        ret self.>age
    }
    
    // Method with parameters and return value
    fn i32 addAge: i32 amount {
        self.>age = self.>age + amount
        ret self.>age
    }
}
```

### Method Calls

Call methods using dot notation on struct instances:

```tc
fn void main: {
    Person person
    person.age = 25
    person.name = "Alice"
    
    // Method calls
    person.sayHi()
    person.setAge(30)
    i32 currentAge = person.getAge()
    i32 newAge = person.addAge(5)
    
    printi(currentAge)  // Prints 30
    printi(newAge)      // Prints 35
}
```

### Self Parameter

Inside methods, `self` refers to the struct instance and is automatically passed as a pointer. Use `.>` syntax to access fields through `self`:

```tc
fn void example: {
    // self is a pointer to the struct instance
    self.>field = value  // Correct: access field through pointer
    // self.field = value  // Error: self is a pointer, not a value
}
```

### Generated C Code

Methods are compiled to C functions with the struct pointer as the first parameter:

```tc
// TightC method
fn i32 getAge: {
    ret self.>age
}

// Generated C function
int32_t Person_getAge(struct Person *self) {
    return self->age;
}
```

### Method Call Translation

Method calls are translated to function calls with the struct address:

```tc
// TightC method call
person.getAge()

// Generated C function call
Person_getAge(&person)
```

### Method Naming Convention

Generated C functions follow the pattern: `StructName_methodName`

- `person.sayHi()` ‚Üí `Person_sayHi(&person)`
- `data.process()` ‚Üí `Data_process(&data)`

### Method Forward Declarations

The compiler automatically generates forward declarations for all methods:

```c
// Generated forward declarations
void Person_sayHi(struct Person *self);
void Person_setAge(struct Person *self, int32_t newAge);
int32_t Person_getAge(struct Person *self);
int32_t Person_addAge(struct Person *self, int32_t amount);
```

### Limitations

- Methods can only be defined inside struns
- Method names must be unique within a strun (no conflicts with field names)
- Methods cannot be static or virtual (they're always instance methods)
- The `self` parameter is implicit and cannot be declared explicitly
- Method overloading is not supported

### Complete Example

```tc
use "stdlib/io.tc"

strun Counter {
    i32 value
    
    fn void init: {
        self.>value = 0
    }
    
    fn void increment: {
        self.>value = self.>value + 1
    }
    
    fn void add: i32 amount {
        self.>value = self.>value + amount
    }
    
    fn i32 getValue: {
        ret self.>value
    }
    
    fn void reset: {
        self.>value = 0
    }
}

fn void main: {
    Counter counter
    counter.init()
    
    counter.increment()
    counter.add(5)
    printi(counter.getValue())  // Prints 6
    
    counter.reset()
    printi(counter.getValue())  // Prints 0
}
```

## Inline C Code

Tig provides an escape hatch for when you need to drop down to C code directly. This is useful for accessing system APIs, performance-critical code, or when TightC doesn't yet support a specific feature.

### Syntax

```tc
"C"{
    // Direct C code here
}
```

### Basic Usage

```tc
use "stdlib/io.tc"

fn void main: {
    print("About to use inline C\n")
    
    "C"{
        printf("Hello from inline C!\n");
        int x = 42;
        printf("C variable x = %d\n", x);
    }
    
    print("Back to TightC\n")
}
```

### Accessing TightC Variables

TightC variables are accessible in inline C blocks:

```tc
fn void main: {
    i32 tc_var = 100
    
    "C"{
        printf("TightC variable value: %d\n", tc_var);
        tc_var = 200;  // Modify TightC variable
    }
    
    printi(tc_var)  // Prints 200
}
```

### Complex C Code

You can define functions, use control flow, and write complex C code:

```tc
fn void main: {
    i32 counter = 0
    
    "C"{
        // Define a function
        void custom_function(int value) {
            printf("Custom function called with: %d\n", value);
        }
        
        // Use loops and conditionals
        for(int i = 0; i < 3; i++) {
            if(i % 2 == 0) {
                printf("Even iteration: %d\n", i);
            } else {
                printf("Odd iteration: %d\n", i);
            }
            counter++;  // Modify TightC variable
        }
        
        custom_function(42);
    }
    
    print("Final counter: ")
    printi(counter)
}
```

### Generated C Code

Inline C blocks are emitted directly into the generated C code:

```tc
// TightC code
fn void main: {
    i32 value = 10
    
    "C"{
        printf("Value is: %d\n", value);
    }
}
```

```c
// Generated C code
void main(void) {
    int32_t value = 10;

    printf("Value is: %d\n", value);
}
```

### Use Cases

**System Calls:**
```tc
"C"{
    #include <unistd.h>
    write(STDOUT_FILENO, "Hello from system call\n", 26);
}
```

**Performance-Critical Code:**
```tc
"C"{
    // Optimized memcpy implementation
    void* fast_memcpy(void* dest, const void* src, size_t n) {
        char* d = (char*)dest;
        const char* s = (const char*)src;
        while(n--) *d++ = *s++;
        return dest;
    }
}
```

**External Libraries:**
```tc
"C"{
    #include <curl/curl.h>
    CURL* curl = curl_easy_init();
    if(curl) {
        curl_easy_setopt(curl, CURLOPT_URL, "https://example.com");
        curl_easy_perform(curl);
        curl_easy_cleanup(curl);
    }
}
```

### Limitations

- Inline C code is not parsed or validated by Tig
- Type safety is not enforced for inline C code
- Debugging information may be less helpful for inline C
- Inline C blocks cannot contain nested `"C"{ ... }` blocks
- Comments in inline C should use C-style `/* */` or `//` syntax

### Best Practices

1. **Keep it minimal** - Use inline C sparingly and for small, focused code blocks
2. **Document clearly** - Add comments explaining why inline C is necessary
3. **Test thoroughly** - Inline C code bypasses some Tig safety checks
4. **Consider alternatives** - Check if the feature can be implemented in Tig first

### Complete Example

```tc
use "stdlib/io.tc"

fn void main: {
    print("Inline C demonstration\n")
    
    i32 result = 0
    
    "C"{
        // Complex calculation using C math library
        #include <math.h>
        
        double angle = 45.0 * M_PI / 180.0;
        double sine_val = sin(angle);
        
        printf("sin(45¬∞) = %f\n", sine_val);
        
        // Store result in TightC variable
        result = (int32_t)(sine_val * 1000);
    }
    
    print("Result scaled by 1000: ")
    printi(result)
    
    // Custom string manipulation
    "C"{
        char buffer[100];
        snprintf(buffer, sizeof(buffer), "Result: %d", result);
        printf("Formatted string: %s\n", buffer);
    }
}
```

## Macros (v1.2.2)

TightC supports text replacement macros that allow you to customize the language syntax. Macros are processed at compile time and expand inline at the point of use.

### Basic Macros

Simple text replacement without parameters:

```tc
# int{i32}
# float{f32}
# char{u8}

fn void main: {
    int x = 10          // expands to: i32 x = 10
    float y = 3.14      // expands to: f32 y = 3.14
    char c = 65         // expands to: u8 c = 65
}
```

Syntax: `# name{replacement}`

### Parametric Macros

Macros with parameters for more flexible replacements:

#### Single-line Parametric Macros

```tc
# twice(x){x * 2}
# square(x){x * x}
# add(a, b){a + b}

fn void main: {
    i32 doubled = twice(5)      // expands to: i32 doubled = 5 * 2
    i32 squared = square(4)     // expands to: i32 squared = 4 * 4
    i32 sum = add(3, 7)         // expands to: i32 sum = 3 + 7
}
```

Syntax: `# name(param1, param2, ...){replacement}`

#### Multi-line Parametric Macros

For more complex replacements spanning multiple statements:

```tc
# swap(T, x, y){
    T temp = x
    x = y
    y = temp
}

fn void main: {
    i32 a = 1
    i32 b = 2
    swap(i32, a, b)             // expands to all three statements
}
```

Parameters are replaced by argument text at each occurrence in the macro body.

### Macro Rules

1. **Text Replacement**: Macros perform pure text substitution. Arguments are inserted literally at each parameter occurrence.

2. **Parameter Matching**: Parameter names must be complete identifiers. In `# twice(x){x * 2}`, only the standalone `x` is replaced, not `x` within other identifiers.

3. **Scope**: Macros are global within a file. They can be defined at the top level and used anywhere after their definition.

4. **Nesting**: Macros can be used within other macros. The compiler expands macros recursively.

5. **Inline C Interaction**: Macros can be used inside inline C blocks and vice versa.

### Example: Complete Macro Usage

```tc
// Define printi using inline C
"C"{
    static void printi(int32_t x) { printf("%d\n", x); }
}

// Basic type aliases
# int{i32}
# float{f32}

// Arithmetic macros
# twice(x){x * 2}
# add(a, b){a + b}

// Multi-line swap macro
# swap(T, x, y){
    T temp = x
    x = y
    y = temp
}

fn void main: {
    int a = 10
    int b = 20
    
    swap(int, a, b)
    printi(twice(add(a, b)))    // prints 60
}
```
