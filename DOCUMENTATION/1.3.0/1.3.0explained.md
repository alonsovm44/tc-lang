# Tig 1.3.0 Features Explained

This document explains all the new features introduced in Tig 1.3.0, designed for someone new to the language.

## Overview

Tig 1.3.0 introduces **concurrency and ownership** features, making Tig a systems programming language with safe memory management and powerful async capabilities. The main additions are:

- **Async functions** - Run code concurrently
- **Queues and Stacks** - Communicate between concurrent tasks
- **Ownership system** - Track who owns data and prevent data races
- **Select statements** - Wait on multiple channel operations

---

## Async Functions

### What are async functions?

Async functions allow you to run code concurrently (in parallel with other code). When you call an async function, it doesn't block - it starts running in the background while your program continues.

### Syntax

```tig
async fn void myAsyncFunction: {
    // This code runs concurrently
}
```

### Key Rules

- **Async functions cannot return values** (Error E012)
- They run in separate goroutines (lightweight threads)
- Perfect for I/O operations, background processing, etc.

### Example

```tig
// Normal function

fn returnType functionName: parameterType parameterName {
    // this runs synchronously
}

// Async function (cannot return values)
async fn void downloadFile: string url {
    // Download file in background
    print("Downloading...")
}

fn void main: {
    async downloadFile("https://example.com/file.txt")
    print("Main continues while download runs")
}
```

## The @ Operator (Ownership Transfer)

### What is ownership?

Tig tracks who "owns" each variable. Only the owner can use the variable. When you transfer ownership with `@`, the original variable becomes "dead" and can't be used anymore.

### Why is this useful?

It prevents data races, two parts of your program can't accidentally modify the same data at the same time.

### Syntax

```tig
@variableName
```

### Key Rules

- **Cannot give owned variable without @** (Error E011)
- After transferring ownership, the original variable is dead
- **Cannot access dead variable** (Error E010)

### Example

```tig
async fn void processData: ->string data {
    print(->data)
}

fn void main: {
    string myData = "important data"
    ->string ptr = &myData
    async processData(@ptr)  // Transfer ownership
    // ptr is now dead - using it here would cause E010
}
```

## Queues and Stacks

### What are queues and stacks?

Queues and stacks are data structures that let different parts of your program send data to each other safely. They're the primary way async functions communicate.

- **Queues** are FIFO (First In, First Out) - like a line at a store
- **Stacks** are LIFO (Last In, First Out) - like a stack of plates

### Syntax

```tig
// Queue (FIFO)
queue i32 q = {1, 2, 3}

// Buffered queue with capacity 10
queue i32(10) buffered_q

// Stack (LIFO)
stack i32 s = {1, 2, 3}
```

### Operations

**Push** (add data):
```tig
queue.push(value)
stack.push(value)
```

**Pop** (get data):
```tig
value = queue.pop()  // Blocks if empty in async context
value = stack.pop()   // Blocks if empty in async context
```

### Example

```tig
async fn void producer: queue i32 q {
    q.push(42)  // Send value
}

async fn void consumer: queue i32 q {
    i32 value = q.pop()  // Receive value
    printi(value)
}

fn void main: {
    queue i32 q
    async producer(q)
    async consumer(q)
}
```

### Multiple Return Values with Queues

Since queues can hold multiple values, you can use them to simulate functions that return multiple values:

```tig
// Function that "returns" multiple values via a queue
async fn void calculateStats: i32[] numbers, queue i32 results {
    i32 sum = 0
    i32 count = 0
    i32 max = numbers[0]
    
    i32 i = 0
    loop if (i < 5) {
        sum = sum + numbers[i]
        if (numbers[i] > max) {
            max = numbers[i]
        }
        count = count + 1
        i++
    }
    
    results.push(sum)    // First "return value"
    results.push(count)  // Second "return value" 
    results.push(max)    // Third "return value"
}

fn void main: {
    i32 nums[5] = [10, 20, 5, 15, 25]
    queue i32 results
    
    async calculateStats(nums, results)
    
    i32 sum = results.pop()   // Get first result
    i32 count = results.pop() // Get second result
    i32 max = results.pop()   // Get third result
    
    print("Sum: ")
    printi(sum)
    print("\nCount: ")
    printi(count)
    print("\nMax: ")
    printi(max)
    print("\n")
}
```

**Benefits of this approach:**
- **Clear ordering**: Results come out in the same order you push them
- **Type safe**: Each queue holds only one type, so all "return values" must be the same type
- **Async friendly**: Works naturally with async functions
- **Extensible**: Easy to add more "return values" by pushing more data

## Select Statements

### What is select?

A `select` statement waits on multiple queue/stack operations at once. It runs whichever one becomes ready first. This is essential for coordinating between multiple data structures.

### Syntax

```tig
select {
    queue1.pop() => {
        value = queue1.pop()
        // Handle receive from queue1
    }
    queue2.push(value) => {
        // Handle send to queue2
    }
    // ... more cases
}
```

### Example

```tig
async fn void worker: queue i32 q1, queue string q2 {
    select {
        q1.pop() => {
            i32 num = q1.pop()
            print("Got number: ")
            printi(num)
            print("\n")
        }
        q2.pop() => {
            string str = q2.pop()
            print("Got string: ")
            print(str)
            print("\n")
        }
    }
}
```

## Pin Analysis

### What is pinning?

Pinning prevents a variable from being modified. This is useful when you need to guarantee that data won't change while you're using it.

### Key Rules

- **Cannot assign to pinned variable** (Error E013)
- Pinned variables are immutable in the current scope

### Example

```tig
async fn void reader1: ->string data {
    print("Reader 1: ")
    print(->data)
    print("\n")
}

async fn void reader2: ->string data {
    print("Reader 2: ")
    print(->data)
    print("\n")
}

fn void main: {
    string myData = "shared data"
    ->string ptr = &myData
    pin ptr  // Make ptr read-only
    
    async reader1(ptr)  // Safe to share
    async reader2(ptr)  // Safe to share
    
    // ptr = &"new data"  // ERROR: Cannot assign to pinned variable
}
```

## Type Categorization

Tig categorizes types into three categories:

1. **Value types** - Primitives like `i32`, `i64`, `f32`, `f64`, `bool`. These are copied when passed around.
2. **Reference types** - Queues and stacks (`queue T`, `stack T`). These refer to thread-safe data elsewhere.
3. **Owned types** - Pointers (`->T`, `=>T`) and structs containing pointers.

This categorization helps the compiler understand how to handle data safely in concurrent contexts.

---

## Error Codes

Tig 1.3.0 introduces these new error codes:

- **E010**: Cannot access dead variable - You tried to use a variable after transferring its ownership
- **E011**: Cannot give owned variable without @ - You need to explicitly transfer ownership with @
- **E012**: Async functions cannot return values - Async functions must return void
- **E013**: Cannot assign to pinned variable - Pinned variables are immutable

## Runtime Features

### Goroutine Runtime

Tig implements a thread pool or similar mechanism to run async functions efficiently. Goroutines are lightweight - you can spawn thousands without performance issues.

### Channel Runtime

Channels are implemented as thread-safe queues, ensuring that concurrent sends and receives don't corrupt data.

## Common Patterns

### Loop Concurrency Pattern

Running multiple iterations of a loop concurrently:

```tig
async fn void processItem: i32 item, queue i32 results {
    print("Processing: ")
    printi(item)
    print("\n")
    results.push(item * 2)  // Send result back
}

fn void main: {
    i32 items[5] = [1, 2, 3, 4, 5]
    queue i32 results
    
    i32 i = 0
    loop if (i < 5) {
        async processItem(items[i], results)
        i++
    }
    
    // Collect results
    i32 j = 0
    loop if (j < 5) {
        i32 result = results.pop()
        print("Result: ")
        printi(result)
        print("\n")
        j++
    }
}
```

### Fan-out/Fan-in

Multiple producers sending to one queue (fan-in):

```tig
async fn void producer: i32 id, queue i32 q {
    i32 i = 0
    loop if (i < 10) {
        q.push(id * 100 + i)
        i++
    }
}

async fn void consumer: queue i32 q {
    i32 i = 0
    loop if (i < 30) {
        i32 value = q.pop()
        print("Received: ")
        printi(value)
        print("\n")
        i++
    }
}

fn void main: {
    queue i32(100) q
    async producer(1, q)
    async producer(2, q)
    async producer(3, q)
    async consumer(q)
}
```

### Fan-out Pattern

One producer sending to multiple queues:

```tig
async fn void distributor: queue i32 q1, queue i32 q2, queue i32 q3 {
    i32 i = 0
    loop if (i < 10) {
        q1.push(i)      // Distribute work
        q2.push(i * 10)
        q3.push(i * 100)
        i++
    }
}

async fn void worker: i32 id, queue i32 q {
    i32 i = 0
    loop if (i < 10) {
        i32 value = q.pop()
        print("Worker ")
        printi(id)
        print(" got: ")
        printi(value)
        print("\n")
        i++
    }
}

fn void main: {
    queue i32 q1, q2, q3
    
    async distributor(q1, q2, q3)  // One producer
    async worker(1, q1)            // Multiple consumers
    async worker(2, q2)
    async worker(3, q3)
}
```

## Summary

Tig 1.3.0 adds powerful concurrency features while maintaining safety through:

- **Async functions** for concurrent execution
- **Queues and Stacks** for safe communication
- **Ownership tracking** via the @ operator
- **Select statements** for coordinating multiple queues/stacks
- **Pin analysis** for immutability guarantees

These features make Tig suitable for systems programming where performance and safety are both critical.
