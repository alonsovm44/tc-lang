# Tight-C 1.2.0 Full Documentation

## Overview

Tight-C 1.2.0 introduces several major features that significantly enhance the language's capabilities for systems programming:

- **Varargs support** — Functions can now declare and use variadic arguments with `...`
- **Expanded I/O stdlib** — Added comprehensive file I/O functions
- **Else if statements** — Added `else if` and `else` for chained conditionals
- **Enums** — Simple enums for type-safe constants
- **Match statements** — Pattern matching with wildcard support
- **Pointer field access** — `.>` syntax for accessing fields through pointers
- **Pass-by-reference** — `&` operator for passing pointers to functions

---

## Varargs Support

Functions can now declare variadic arguments using `...`:

```tc
extern "C" {
    i32 fn printf: ->i8 fmt, ... {}
}
```

### Usage

When calling varargs functions, you don't need to write `...` in the call - it's implicit:

```tc
use "stdlib/io.tc"

extern "C" {
    i32 fn printf: ->i8 fmt, ... {}
}

fn void main: {
    printf("Hello %s, number: %d\n", "world", 42)
}
```

### Implementation Notes

- `...` in function declarations is emitted as `...` in the generated C function signature
- `...` in function calls is not emitted - varargs are implicit in C function calls
- The compiler tracks varargs with the `EX_VARARGS` AST node

## Expanded I/O Stdlib

### Standard I/O Functions

| Function | Description |
|----------|-------------|
| `print(s)` | Print string + newline |
| `printn(s)` | Print string, no newline |
| `printi(n)` | Print i64 + newline |
| `printin(n)` | Print i64, no newline |
| `readi()` | Read i64 from stdin |
| `readc()` | Read single char from stdin |
| `unreadc(c, stream)` | Push char back to file stream |
| `write_file(s, stream)` | Write string to file |
| `eof(stream)` | Check if at end of file |

### File I/O (via extern "C")

| Function | Description |
|----------|-------------|
| `fopen(file, mode)` | Open file |
| `fclose(f)` | Close file |
| `fgetc(stream)` | Read char from file |
| `ungetc(c, stream)` | Push char back to file |
| `fputs(s, stream)` | Write string to file |
| `fprintf(stream, fmt, ...)` | Formatted print to file |
| `fscanf(stream, fmt, ...)` | Formatted read from file |
| `feof(stream)` | Check end of file |

### Example

```tc
use "stdlib/io.tc"

extern "C" {
    ->FILE fn fopen: ->i8 file, ->i8 mode {}
    i32 fn fclose: ->FILE stream {}
    i32 fn fputs: ->i8 s, ->FILE stream {}
}

fn void main: {
    ->FILE f = fopen("test.txt", "w")
    if (f) {
        fputs("Hello, file!\n", f)
        fclose(f)
    }
}
```

## Else If Statements

Tight-C 1.2.0 introduces `else if` for chained conditionals, with `else` as the final branch:

```tc
fn void check_grade: i32 score {
    if (score >= 90) {
        print("A")
    } else if (score >= 80) {
        print("B")
    } else if (score >= 70) {
        print("C")
    } else if (score >= 60) {
        print("D")
    } else {
        print("F")
    }
}
```

### Syntax Rules

- `else if (condition) { ... }` — else-if branch
- `else { ... }` — final else branch (must be last)
- The `_` wildcard is now reserved for match statements only

## Enums

Tight-C 1.2.0 introduces simple enums for type-safe constants:

```tc
enum Color {
    RED
    GREEN
    BLUE
    BLACK
    WHITE
}

fn void main: {
    Color c = RED
    if (c == RED) {
        print("Color is RED\n")
    }
}
```

### Syntax Rules

- `enum Name { MEMBER1, MEMBER2, ... }` — define an enum
- Enum members are scoped to the enum type
- Types default to i32 (C doesn't support enum type specification like C++)
- Generated as `typedef enum { ... } Name;` in C

### Usage

Enums can be used as variable types and in comparisons:

```tc
enum Status {
    ACTIVE
    INACTIVE
    PENDING
}

fn void set_status: Status s {
    if (s == ACTIVE) {
        print("Status is active\n")
    }
}
```

## Match Statements

Pattern matching with wildcard support:

```tc
fn void describe: i32 n {
    match (n) {
        1 = {
            print("one")
        }
        2 = {
            print("two")
        }
        3 = {
            print("three")
        }
        _ = {
            print("other")
        }
    }
}
```

### Syntax

- `match (expr) { ... }` — match expression against patterns
- `pattern = { ... }` — pattern branch
- `_ = { ... }` — wildcard (catch-all) branch
- Patterns are currently single values (literals or variables)

## Pointer Field Access (.>)

To avoid ambiguity with the `->` operator (used for pointer declaration and dereference), Tight-C uses `.>` for accessing fields through pointers:

```tc
strun Person {
    i8 age
    ->i8 name
}

void fn foo: ->Person p {
    p.>age = 23
    p.>name = "John"
}

void fn main: {
    Person p
    foo(&p)
    printi(p.age)
    print(p.name)
}
```

### Generated C

```c
void foo(struct Person *p) {
    p->age = 23;
    p->name = "John";
}

void main(void) {
    struct Person p = {0};
    foo(&p);
    printi(p.age);
    print(p.name);
}
```

### Alternative Syntax

You can also dereference explicitly:

```tc
void fn boo: ->Person p {
    (->p).age = 23
    (->p).name = "John"
}
```

## Pass-by-Reference (&)

Use `&` to pass pointers to functions:

```tc
void fn modify: ->i32 x {
    ->x = 100
}

fn void main: {
    i32 value = 10
    modify(&value)
    printi(value)  // Prints 100
}
```

### Generated C

```c
void modify(int32_t *x) {
    *x = 100;
}

void main(void) {
    int32_t value = 10;
    modify(&value);
    printi(value);
}
```

## Hot Function Reloading

Hot reloading allows you to modify code and recompile shared libraries without restarting the main executable.

### Basic Usage

```tc
use "stdlib/io.tc"

hot fn i32 add: i32 x, i32 y {
    ret x + y
}

fn void main: {
    loop {
        i32 result = add(3, 4)
        printi(result)
    }
}
```

Compile with hot reload enabled:
```bash
tightc source.tc -H hotlib -c app
```

Run the application:
```bash
./app
```

While running, modify the hot function and rebuild:
```bash
tightc source.tc -H hotlib --hot-rebuild
```

The running application automatically detects the change and reloads the library.

### How It Works

Hot reload uses versioned shared libraries to avoid file locking issues on Windows:

- Functions marked with `hot` are compiled into a separate shared library
- Each rebuild creates a new version (e.g., `hotlib_1.dll`, `hotlib_2.dll`)
- A version file tracks the current version number
- The host executable monitors the version file and dynamically loads new versions

### Additional Flags

| Flag | Description |
|------|-------------|
| `-H <libname>` | Enable hot reload mode, specify the library name |
| `--hot-rebuild` | Rebuild only the hot library (for running applications) |
| `-t, --temp` | Keep temporary .c files for debugging |

## Compiler Usage

```bash
tightc <input.tc> [-o output.c] [-c binary]
```

| Flag | Description |
|------|-------------|
| `-o file.c` | Emit transpiled C to file (`.h` gets `#pragma once`) |
| `-c binary` | Transpile + compile to binary (auto-detects gcc/clang) |
| `-H <libname>` | Enable hot reload mode |
| `--hot-rebuild` | Rebuild only hot library |
| (none) | Print transpiled C to stdout |

Combine both: `tightc app.tc -o app.c -c app` keeps the `.c` and builds the binary.

## Strunions (Layout Polymorphism)

Strunions combine structs and unions into a single construct:

```tc
// Regular struct
strun Point {
    i32 x
    i32 y
}

// Union
strun Data {
    &i32 data
    &str ip
}

// Hybrid
strun Hybrid {
    i32 x
    i32 y
    &i32 z
    &f32 w
}
```

- Regular fields -> struct behavior (separate memory)
- `&`-prefixed fields -> union behavior (shared memory)
- Anonymous padding for alignment control
